# AirPlayer 产品规划与可行性分析（v0.2.0 之后）

> 双视角：**产品经理**给想法与价值，**开发者**给可行性、工作量与落地路径。
> 工作量 S/M/L = 小/中/大；可行性以当前架构（D3D11 解码纹理管线 + waveOut 音频 + SessionManager 会话）为依据。

现状一句话：能稳定接收 iPhone/iPad 屏幕镜像，视频全 GPU 硬解、音频 AAC-ELD、音画同步；但 UI 仅一个播放窗口，缺设置、缺常驻、缺录制等"产品化"能力。

---

## 一、体验 / UI

### 1. 待机欢迎页（连接前引导）— 价值高 / 工作量 S
- **PM**：没连接时是黑屏，用户不知道是否就绪。做一个待机页，显示本机 AirPlay 名称、局域网 IP、"在 iPhone 控制中心 → 屏幕镜像 选择 XXX"的图文步骤，连接成功自动切到画面。
- **Dev**：易落地。`AirPlayReceiver` 已知广播名；新增一个 XAML 覆盖层，订阅"连接建立/断开"事件切换可见性。无协议改动。

### 2. 悬浮控制条 — 价值高 / 工作量 S–M
- **PM**：鼠标移动时浮现一排按钮：全屏、旋转、截图、录制、设置；3 秒不动自动隐藏。
- **Dev**：易。WinUI 顶层 `Grid` + 自动隐藏定时器；旋转已有 `NotifyRotation`，其余按钮接到下方功能。

### 3. 设置面板 + 配置持久化 — 价值高 / 工作量 M
- **PM**：可调：缩放模式（信箱/填充/原始）、音频输出设备、低延迟模式、窗口置顶、无边框、AirPlay 名称、主题。
- **Dev**：中。新增设置页 + JSON 配置存 `%LocalAppData%\AirPlayer\settings.json`；多数项映射到现有管线参数（缩放在 `Nv12VideoProcessor`、置顶/无边框是窗口属性、名称在 mDNS 广播）。

### 4. 系统托盘 + 开机自启 + 最小化到托盘 — 价值中 / 工作量 M
- **PM**：当作常驻"无线副屏"，关窗即最小化到托盘，开机自启。
- **Dev**：中。WinUI3 无原生托盘，引入 `H.NotifyIcon.WinUI` 或 P/Invoke `Shell_NotifyIcon`；自启写注册表 `Run` 键或快捷方式。

### 5. 实时 HUD（FPS / 分辨率 / 码率 / 延迟 / 丢帧）— 价值中 / 工作量 S
- **PM**：发烧友与排障都需要；可热键开关。
- **Dev**：易。`VideoPresenter` 已有解码/呈现/丢帧计数，叠加一个 `TextBlock` 即可；默认关闭。

### 6. 现代化窗口与深浅色主题 — 价值中 / 工作量 S
- **Dev**：易。WinUI 主题资源 + Mica/亚克力标题栏。

## 二、功能

### 7. 录屏到 MP4（含音频）— 价值最高 / 工作量 M
- **PM**：把投屏录成文件，是这类工具的"杀手级"诉求（教学、演示、游戏录制）。
- **Dev**：中，且我们占优势——管线里**已有解码前的 H.264 原始帧、解码后的 NV12 纹理、PCM 音频**。两条路：
  - **(a) 直接 remux 原始流**（推荐起步）：把收到的 H.264 帧与音频按 MP4 封装，**几乎零再编码、零 GPU 开销**。难点是音频：ELD 不是标准 MP4 音轨，需转封装为 PCM/AAC-LC，或录制时把已解码 PCM 用 MF 编 AAC-LC 再封装。
  - **(b) MF SinkWriter 全编码**：NV12→H.264(GPU) + PCM→AAC，通用但有编码开销。
  - 建议：先做"录视频(remux H.264) + 录音频(PCM→AAC-LC)"，用 Media Foundation `IMFSinkWriter` 封装 MP4。

### 8. 截图（PNG）— 价值中 / 工作量 S
- **Dev**：易。把后台缓冲或 NV12 纹理拷到 `Staging` 纹理 → WIC 存 PNG。一个按钮 + 热键。

### 9. 音频输出设备选择 + 切到 WASAPI 低延迟 — 价值高 / 工作量 M
- **PM**：当前固定系统默认设备，且 `waveOut` 延迟偏高、设备不可选。想投到指定音箱/耳机，并降低音频延迟。
- **Dev**：中。先做 `waveOut` 设备枚举（`waveOutGetNumDevs`）快速版；进阶改 **WASAPI 共享/独占模式**，延迟更低、音质更好，也便于做设备热切换。替换点集中在 `AudioSink`。

### 10. 自动旋转 — 价值中 / 工作量 M
- **PM**：手机横竖屏切换时画面自动转正（现在要手动点旋转）。
- **Dev**：中。iOS 旋转会发新 SPS/分辨率，管线已能感知宽高变化；据宽高比变化自动调用 `NotifyRotation`。需实测：部分镜像流本身已按方向编码，避免重复旋转。

### 11. 连接密码 / 已配对设备白名单 — 价值中 / 工作量 M
- **PM**：同一 Wi-Fi 下防止他人随意投屏。
- **Dev**：中。AirTunes 已有 pair-setup 配对流程；加 RTSP 密码校验，或仅"记住已配对设备"白名单，陌生设备需确认。

### 12. 多设备 / 快速切换 — 价值中 / 工作量 L
- **PM**：列出当前与历史设备，一键切换。
- **Dev**：较大。`SessionManager` 已按 client 管理会话，但解码器/交换链目前面向单流；多流需资源复用或按设备重建，UI 也要会话列表。

### 13. 纯音频模式（AirTunes 音乐投放）— 价值中 / 工作量 M
- **PM**：把手机音乐投到 PC 音箱（不投屏）。
- **Dev**：中。代码已有 AAC-LC（AAC-main）分支，但走的微软 MFT 在部分机器未注册；可统一用 WASAPI 输出 + 现有解码路径，UI 区分"镜像/纯音频"两类会话。

## 三、底层 / 质量

### 14. PTS 主时钟精确同步 — 价值中 / 工作量 M–L
- **PM**：当前靠"音视频都压到低延迟"实现对齐，绝大多数场景够用；要更严格的唇音同步可升级。
- **Dev**：中偏大。`AudioSink.CurrentClock` 已是音频时钟；把 `VideoPresenter` 从"始终呈现最新帧"改为"按 PTS 调度呈现"，需引入显示缓冲与丢帧策略，回归风险中等，建议有需求再做。

### 15. 设备丢失自动重建管线 — 价值中 / 工作量 M
- **Dev**：`VideoPresenter` 已检测 `DXGI_ERROR_DEVICE_RESET`（目前仅日志）。补上 UI 线程重建交换链/解码器的恢复流程，提升长时间运行稳健性。

### 16. 安装包 / 代码签名 / 自动更新 — 价值高（正式版前必做）/ 工作量 M
- **Dev**：MSIX 或 Inno Setup 做安装器；代码签名证书；启动时检查版本更新。配合已有 `build-release.ps1`。

## 四、建议路线图

| 版本 | 主题 | 包含 | 风险 |
|---|---|---|---|
| **v0.3** | 体验打底（低风险快赢） | 待机欢迎页(1)、悬浮控制条(2)、HUD(5)、截图(8)、设置+持久化(3 起步) | 低 |
| **v0.4** | 核心功能 | 录屏 MP4(7)、音频设备选择/WASAPI(9)、自动旋转(10)、托盘+自启(4) | 中 |
| **v0.5** | 进阶 | 连接密码(11)、纯音频模式(13)、设备丢失重建(15)、PTS 精确同步(14) | 中 |
| **v1.0** | 正式发布 | 多设备(12)、安装包/签名/更新(16)、设置与文档完善 | 中 |

**优先级建议**：先把 v0.3 的"待机页 + 控制条 + 截图"做掉——工作量小、立刻让产品"像个成品"；同时启动 v0.4 的**录屏**（差异化最强、技术上我们有原始流优势）与**WASAPI 音频**（顺手解决延迟与设备选择两件事）。
