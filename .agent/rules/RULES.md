# 代码注释添加规则

1. 原则上，尽量为每一行代码添加中文注释。
2. 注释内容要简洁、准确，直接说明该行代码的作用。
3. 所有注释统一使用中文，不使用英文注释。
4. 若某行是空行、括号行或语法分隔行，则不添加注释。
5. 在不破坏语法的前提下，优先使用行内注释；若语言不支持行内注释，则使用上一行中文注释。
6. 定义函数时，函数定义行可以只写用途注释，参数说明优先放在 docstring 中统一说明参数、返回值和用途。

示例：

```python
x = 10  # 定义变量x并赋值为10
y = 20  # 定义变量y并赋值为20
z = x + y  # 计算x与y的和并赋值给z
print(z)  # 输出z的值
def __init__(self, scaler, fit_scaler=False):  # 定义初始化方法
	"""
	scaler: DataScaler 实例
	fit_scaler: 是否在当前数据上拟合归一化器
	"""
	if fit_scaler:  # 判断是否需要拟合归一化器
		self.scaler = scaler  # 保存归一化器实例
	else:  # 处理不拟合归一化器的情况
		self.scaler = scaler  # 直接保存归一化器实例
```

# 代码文件命名规则

1. 文件名应简洁、准确，直接说明功能。
2. 使用英文，使用下划线命名法。
3. 后缀用 v{version} 表示版本号。
4. 如果是小型改动（如修复bug、优化性能等），可以在当前版本上直接修改。
5. 如果是大型改动（如功能新增、架构调整等），则创建新版本的代码文件，旧版本号的文件保留。

# 代码组织规则

1. 优先保持脚本【自包含、单文件】。除非用户明确要求，不要把一个脚本的功能拆分到多个独立模块/文件。
2. 通用的辅助函数（如绘图样式、导出工具）直接【内联】写在使用它的脚本里，方便初学者阅读与管理；宁可少量重复，也不引入需要额外导入/维护的共享模块。
3. 用户是初学者，文件越少、依赖越少越好。

# 代码测试规则

1. 如果需要测试，请新建一个文件夹 test，并将测试文件放入其中来运行。
2. Abaqus脚本的测试可以直接运行，不需要进入Abaqus环境。
