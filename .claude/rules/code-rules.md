# 编码约定

## 代码注释添加规则

1. 为关键代码和配置参数含义添加中文注释。
2. 注释内容要简洁、准确，直接说明该行代码的作用。
3. 所有注释统一使用中文，不使用英文注释。

## Git 版本管理规则

采用 Trunk-based 工作流（单主干 `main`），不设长期 `develop` 分支。

1. **分支策略**：
   - 所有日常开发、功能新增、问题修复和重构直接在 `main` 分支上进行提交。
   - 大型改动或需要隔离的功能可从 `main` 拉出短期 `feature/*` 分支，开发完成快进合并回 `main` 并删除该分支；不保留长期集成分支。
2. **提交信息规范**：
   - 所有提交必须遵守 Conventional Commits 规范，前缀为小写的类型（如 `feat:`, `fix:`, `refactor:`, `chore:` 等），后接半角冒号和空格，描述使用**简体中文**。
     - 格式：`<type>: <中文描述>`
     - 示例：`feat: 更多设置新增视频播放帧率选择限制功能（支持 60fps 与 30fps 切换）`
   - **禁止**以版本号做提交类型前缀（如 `v1.0.0: ...`、`v0.5.0: ...`），这不符合 Conventional Commits 规范。
   - 版本发布/里程碑提交使用 `chore: 发布 v{version}，<一句话描述>`，例如：
     `chore: 发布 v1.1.0，修复检查更新闪退与版本号显示问题`
3. **发版规范**：
   - 发布新版本时在 `main` 当前提交上打标签，标签名即版本号（如 `v1.0.1`），附注标签消息为 `AirPlayer v{version}`。
   - 命令：`git tag -a v{version} -m "AirPlayer v{version}"`，随后 `git push origin main --tags`。
   - 版本号遵循语义化版本（SemVer）：主版本.次版本.修订号。
